# `@kbml-tentacles/vue`

Vue 3 adapter for Tentacles. Wires `ViewModelDefinition` and `Model` from `@kbml-tentacles/core` into Vue's composition API. Props are supplied as getters (preserving reactivity); component lifecycle maps onto `lifecycle.mount`/`lifecycle.destroy`. Effector units are flagged with `markRaw` so Vue's reactivity system does not proxy their internals.

— *Reference · Vue adapter · v0.x*

## Install

```bash
yarn add @kbml-tentacles/vue @kbml-tentacles/core effector effector-vue
```

## Peer dependencies

| Package | Range |
|---|---|
| `effector` | `^23.0.0` |

| `@kbml-tentacles/core` | `workspace:*` |

Vue 3.3+ is required for the typed `defineProps`/`defineEmits` and the `InjectionKey` semantics used by `ScopeStackKey`.

## Exports

| Symbol | Kind | Purpose |
|---|---|---|
| [`View`](/reference/vue/view) | component | **Primary way to mount a view-model.** Provides its shape to descendants via `provide`/`inject`. |
| [`Each`](/reference/vue/each) | component | Render a list of model instances, passing each as a scoped-slot argument. |
| [`useModel`](/reference/vue/use-model) | composable | Read a view-model shape from `<View>`, an `<Each>` instance, or look up a model by id. |
| [`useView`](/reference/vue/use-view) | composable | Single-component alternative to `<View>` for owners with no descendants. |
| `ScopeStackKey` | `InjectionKey` | Injection key for the scope stack. Advanced use. |

## Lifecycle contract

| Vue | Core |
|---|---|
| First `setup()` of `<View>` or `useView` | `definition.create(getProps?.())` |
| `onMounted` | `lifecycle.mount()` |
| `watch(getProps, …, { deep: true })` | `definition.applyProps(props)` |
| `onUnmounted` | `lifecycle.destroy()` |

Both `<View>` and `useView` call `destroy` on unmount. Vue does not have a StrictMode unmount/remount cycle, so the React `<View>` unmount-vs-destroy distinction does not apply. Prefer `<View>` — it provides the shape to descendants via `provide`/`inject` so children can pull it with `useModel`. Reach for `useView` only when a single component owns the instance and has no descendants that need `useModel`.

## Prop getter form

`<View>` and `useView` expect props as a **getter function**:

```ts
useView(myViewModel, () => ({ query: props.query, onClear: () => clear() }));
```

The getter is registered with `watch(..., { deep: true })`. Reactive sources inside the getter (Vue refs, computed, props) drive re-application automatically. A plain non-getter object is treated as a one-time initial snapshot and never re-applied.

## Event-to-emit mapping

`useView` accepts an optional `emit` argument (the `defineEmits` emit function). Event-prop names beginning with `on` are re-routed to Vue emits using camelCase → kebab-case:

| Prop name | Emitted event |
|---|---|
| `onClear` | `clear` |
| `onSaveItem` | `save-item` |
| `onSubmitForm` | `submit-form` |
| `onClose` | `close` |

Prop names without a leading `on` are not forwarded; they must be bound explicitly.

## markRaw / toRaw

Effector stores and events are JS objects with internal mutable state. Vue's `reactive()` would wrap them in a Proxy and break referential integrity. The adapter calls `markRaw` on every unit before handing it to the shape, and `toRaw` when reading user-supplied getter props, to prevent Vue from reaching into effector internals.

## Reading effector stores

The adapter does not re-export `useUnit`. Import it from `effector-vue/composition`:

```ts
import { useUnit } from "effector-vue/composition";
import { useModel } from "@kbml-tentacles/vue";

const vm = useModel(counterViewModel);   // reads nearest <View>
const count = useUnit(vm.$count);        // Ref<number>
const increment = useUnit(vm.increment); // function
```

Template reads auto-unwrap the ref: `<span>{{ count }}</span>`.

## SSR

View-models behave identically on the server: `definition.create` runs during setup, `onMounted` is skipped, `onUnmounted` fires when the request scope is disposed. SIDs are generated by core's `SidRegistry` so hydration re-uses the same instance when the client mounts.

The adapter is safe to render under `effector`'s `fork(scope)` and `serialize(scope)` — `<View>` and `<Each>` do not hold references to the default scope.
